# Integration Plan: base_frontend React UI → Python FastAPI Backend

## Problem

The React frontend (`base_frontend/`) was built for Google AI Studio with its own Express server calling Gemini directly. The Python backend (`src/advogado_de_bolso/`) has a superior agent with RAG, deadline calculation, and document drafting tools. They have **mismatched API contracts**:

- **Python backend**: `POST /api/chat` → `{session_id, response}` (plain text)
- **React frontend expects**: `{stepTitle, stepContent, relevantTitle, relevantContent, calcPerformed, deadline, questions, suggestiveText, templateLetter, quickReplies}`

Additionally, the React frontend stores cases only in-memory (React state), while the Python backend has session management but no case persistence.

## Architecture Decision

**Single-server adapter pattern** — the Python FastAPI backend serves everything.

- **Dev**: Vite dev server (port 5173) with proxy to FastAPI (port 8000)
- **Prod**: React builds to static files, FastAPI serves them on port 8000
- **Adapter endpoint** transforms the agent's plain-text response + tool call history into the structured JSON the React UI expects

### Why not alternatives?

| Approach | Verdict | Reason |
|----------|---------|--------|
| Modify Python agent to output structured JSON | ❌ | Breaks Pydantic AI tool-calling; agent needs to be free-form to use tools properly |
| Modify React to accept plain text | ❌ | Loses rich UI features (deadline cards, template letters, quick replies) |
| Keep Express server, proxy to Python | ❌ | Unnecessary extra process; complexity without benefit |
| **Adapter endpoint on FastAPI** | ✅ | Minimal changes; preserves both systems; inspectable tool history |

## Implementation Steps

### Step 1: Create the adapter module

**New file**: `src/advogado_de_bolso/adapter.py`

This module inspects the agent's tool call history (`result.all_messages()`) and the final response text to extract structured data. Pydantic AI's `ModelMessage` objects contain `ToolCallPart` and `ToolReturnPart` that record what tools were called and what they returned.

Key logic:
- Scan `ToolReturnPart` messages for `calcular_prazo_consumidor` output → parse dates and legal basis into a `deadline` object
- Scan `ToolReturnPart` messages for `redigir_documento` output → extract `templateLetter`
- Scan `ToolReturnPart` messages for `search_knowledge_base` output → extract `relevantContent`/`relevantTitle`
- Parse the agent's final response text → extract `stepTitle`, `stepContent`
- Generate `questions` and `quickReplies` based on context patterns

The adapter should define a Pydantic model `StructuredChatResponse` matching the React frontend's expected shape.

### Step 2: Add structured chat method to ChatService

**Modify**: `src/advogado_de_bolso/service.py`

- Add a new method `chat_structured(message, session_id) -> StructuredChatReply` that:
  1. Calls the existing agent via `AgentChatBackend.run()`
  2. Passes the result to the adapter to produce structured output
  3. Returns a `StructuredChatReply` with both `session_id` and the structured fields
- The existing `chat()` method stays unchanged (backward compatible)

### Step 3: Add new API endpoints

**Modify**: `src/advogado_de_bolso/api.py`

#### 3a. Structured chat endpoint
- `POST /api/chat/structured` — accepts `{message, session_id?, response_style?}`, returns `StructuredChatResponse`

#### 3b. Case management endpoints
The React frontend's "cases" map to named sessions. Add:
- `GET /api/cases` — list all cases (sessions with titles and metadata)
- `POST /api/cases` — create a new case (returns case_id / session_id)
- `PUT /api/cases/{case_id}` — rename a case
- `DELETE /api/cases/{case_id}` — delete a case and its session
- `GET /api/cases/{case_id}/history` — get chat history for a case

### Step 4: Add case persistence to ChatService

**Modify**: `src/advogado_de_bolso/service.py`

Currently sessions are in-memory with LRU eviction. For case persistence:
- Add a lightweight persistence layer (JSON file at `./storage/cases.json` or SQLite)
- Store case metadata: id, title, created_at, last_message, icon_name, response_style
- Store chat history per case (the `ModelMessage` list serialized to JSON)
- Keep the existing in-memory session management for active conversations
- Cases that are "open" stay in memory; closed cases are loaded from disk on demand

### Step 5: Serve React static files from FastAPI

**Modify**: `src/advogado_de_bolso/api.py`

- Build the React app: `cd base_frontend && npm run build` → outputs to `base_frontend/dist/`
- Mount `base_frontend/dist/` as static files on FastAPI
- Change the catch-all route from serving the vanilla frontend to serving `base_frontend/dist/index.html`
- The old vanilla frontend at `src/advogado_de_bolso/frontend/` can be kept as a fallback or removed later

### Step 6: Configure Vite dev proxy

**Modify**: `base_frontend/vite.config.ts`

- Add `server.proxy` config to forward `/api/*` requests to `http://localhost:8000`
- This allows the React dev server to talk to the Python backend during development

### Step 7: Update React frontend API calls

**Modify**: `base_frontend/src/App.tsx`

- Change `fetch("/api/chat", ...)` to `fetch("/api/chat/structured", ...)`
- Update the request body to match the new endpoint's expected shape (add `session_id` when available)
- Store the returned `session_id` in state for subsequent requests
- Handle `snake_case` field names from Python (or convert in the adapter to `camelCase`)

### Step 8: Update React frontend case management

**Modify**: `base_frontend/src/App.tsx`

- Replace in-memory `useState<Case[]>` with API calls to `/api/cases`
- `handleSelectCase` → fetch case history from API
- `handleDeleteCase` → call DELETE endpoint
- `handleRenameCase` → call PUT endpoint
- `handleStartConsultation` → call POST to create case, then send first message
- Load cases from API on mount (`useEffect`)
- Remove the hardcoded `seedCases` (or migrate them to the backend)

### Step 9: Update React frontend types

**Modify**: `base_frontend/src/types.ts`

- Add `sessionId` field to `Case` type
- Ensure all fields from `StructuredChatResponse` are covered
- Add any new types for API responses

### Step 10: Add npm build to Python package

**Modify**: `pyproject.toml`

- Add a build hook or script to run `npm install && npm run build` in `base_frontend/` before packaging
- Alternatively, document the build step in README
