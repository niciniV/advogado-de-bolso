# DOX framework

- DOX is highly performant AGENTS.md hierarchy installed here
- Agent must follow DOX instructions across any edits

## Project Overview

Agentic chatbot for Brazilian consumer rights (CDC/Lei 8.078/90). Python 3.11–3.14, Pydantic AI 1.106+ agent, ChromaDB RAG, FastAPI HTTP API, Rich CLI. Implementation follows a 9-batch gated plan (see `.opencode/plans/20-implementation-order.md`); batch 1 introduced typed tool return envelopes in `src/advogado_de_bolso/contracts.py` (DeadlineResult, DraftedDocument, KnowledgeChunk + Tom/TipoPrazo aliases), batch 2 added wire types (`schemas.py`) and the LLM-output → wire-response adapter (`adapter.py`) with golden tests in `tests/test_adapter.py`, batch 3 added per-case JSON persistence (`src/advogado_de_bolso/storage/cases.py`) with `Case` model + `load`/`save`/`delete`/`list_all` and the storage golden tests in `tests/test_storage.py`, **batch 4 rewrote `agent.py` (full merged `SYSTEM_PROMPT` + `STYLE_PROMPTS` + `_current_style` ContextVar + `@agent.instructions`), rewrote `service.py` (disk-persistent `ChatService` with per-case `asyncio.Lock`, `chat_structured` as the new primary method, reviewer-gated persistence of `chat_history` AND `model_history`), rewrote `api.py` (new case-management endpoints, UUID-typed path params, explicit SPA fallback, blocked-envelope 422 contract), added `cases_path` to `Settings` (env alias `CASES_PATH`), and rewrote/extended the four gate test files (`test_config.py`, `test_agent.py`, `test_service.py`, `test_api.py`)**, **batch 5 rewrote `cli.py` (reviewer-gated buffered streaming: stays on `agent.run_stream` per Open Issue #1 = A, accumulates tokens into an internal buffer behind a `rich` spinner, calls `review_response(...)` with `model` and `model_settings` after generation, persists approved turns via `cases.save(case, cases_path=settings.cases_path)`, displays only `REVIEW_BLOCKED_MESSAGE` on blocks) and added `tests/test_cli.py` (19 tests covering the approved/blocked envelope, reviewer kwargs, two-turn UUID reuse, `/limpar` fresh-case-without-overwrite, and disk persistence)**, **batch 6 rewrote `base_frontend/package.json` (dropped the `server.ts`-referencing scripts `dev`/`build`/`start`/`clean`; removed `dependencies` `@google/genai`/`express`/`dotenv`/`motion` and `devDependencies` `tsx`/`esbuild`/`@types/express`; added `vitest`/`jsdom`/`@testing-library/react`/`@testing-library/jest-dom`; KEEPS `@types/node` per ISSUE-USR-014), updated `base_frontend/vite.config.ts` to add the `/api` proxy to `http://localhost:8000` (ISSUE-DS-003) plus the `test: { environment: "jsdom", setupFiles: ["./src/test/setup.ts"], globals: true }` block (imports `defineConfig` from `vitest/config` not `vite`), updated `base_frontend/tsconfig.json` to include `types: ["vitest/globals", "@testing-library/jest-dom"]` and `exclude: ["server.ts"]` (transitional — server.ts is deleted in batch 7), created `base_frontend/src/test/setup.ts` (single line: `import "@testing-library/jest-dom/vitest";`), generated `base_frontend/package-lock.json` via `npm install`, and created `base_frontend/AGENTS.md`** (batch 6 verification: `npm install` 312 packages, `npm run lint` clean, `npm run build` produced `dist/`; `npm run test` correctly reports "No test files found" since tests are added in batch 7), and **batch 7 added the frontend HTTP client (`base_frontend/src/api.ts` with `apiClient` covering `POST /api/chat/structured`, `GET /api/cases`, `GET/PATCH/DELETE /api/cases/{id}`, `GET /api/cases/{id}/history` plus the `WireCaseSummary`/`WireCaseResponse`/`WireChatMessage`/`WireStructuredChatRequest`/`WireStructuredChatResponse` wire types and the `mapStructuredResponse`/`mapCaseSummary`/`deriveLastMessage`/`deriveTagText`/`formatCaseDate` adapters), refactored `base_frontend/src/App.tsx` to wire the API client (loads real cases on mount, falls Stays on `agent.run_stream` (per Open Issue #1 = A); tokens are accumulated into an internal buffer while the user sees a `rich` spinner. After generation completes, `_process_turn` invokes `review_response(...)` with `question`, `response`, `model=settings.full_model_name`, and `model_settings=settings.build_model_settings()`. On approval, the loop appends to `case.chat_history` and `case.model_history`, advances `case.updated_at`, and calls `cases.save(case, cases_path=settings.cases_path)`. On block, the loop renders only `REVIEW_BLOCKED_MESSAGE` and does NOT save. The CLI does NOT use `ChatService` — it persists via the storage layer directly with its own reviewer gate. The `/limpar` slash command creates a fresh in-memory case with a new UUID without overwriting the previously saved case file. Concurrent API+CLI edits to the same case are last-writer-wins (atomic JSON replacement prevents torn JSON, not cross-process lost updates); the limitation is documented in the module docstring. |
