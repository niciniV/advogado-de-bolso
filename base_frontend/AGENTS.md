# base_frontend - React Frontend

## Purpose

React + Vite + TypeScript frontend for "Advogado de Bolso". Single-page application that talks to the FastAPI backend (see `../src/advogado_de_bolso/api.py`) over HTTP. The backend serves the production build from `../base_frontend/dist/`; the Vite dev server proxies `/api/*` to the FastAPI server on port 8000 (per `.opencode/plans/10-frontend-build-and-config.md`, ISSUE-DS-003).

## Ownership

- `vite.config.ts` - Vite + Vitest config (proxy, jsdom test environment, global test setup).
- `tsconfig.json` - TypeScript compiler config (Vitest global types, jsdom matcher augmentations).
- `package.json` - Frontend deps + npm scripts (`dev`, `build`, `preview`, `lint`, `test`, `clean`).
- `package-lock.json` - Reproducible install (committed; the `Makefile` target uses `npm ci`).
- `index.html` - Vite HTML entry.
- `src/main.tsx` - React root mount (`StrictMode` + `createRoot`).
- `src/App.tsx`, `src/components/*.tsx` - Application UI (REFACTORED batch 7: API client integration, `isSendingMessage` rename).
- `src/types.ts` - Shared TypeScript types for `Case`, `ChatMessage`, `Deadline`, etc. (REFACTORED batch 7: adds `is_demo?` to `Case`).
- `src/defaults.ts` - Default cases and preferences (NEW batch 7).
- `src/api.ts` - HTTP client to the FastAPI backend (NEW batch 7).
- `src/api.test.ts`, `src/App.test.tsx` - Vitest tests (NEW batch 7).
- `src/test/setup.ts` - Vitest setup (loads `@testing-library/jest-dom/vitest`; jsdom `scrollIntoView` polyfill added in batch 7).
- `assets/` - Static assets served by Vite.
- `.env.example` - Sample env vars (no `GEMINI_API_KEY` - the FastAPI server reads its own env).

## Local Contracts

- **Build is Vite-only** (batch 6). The previous `server.ts` (Express + `@google/genai`) was **DELETED in batch 7**; the rewritten `package.json` drops `tsx`, `esbuild`, `@types/express`, `express`, `dotenv`, `motion`, and `@google/genai`. The `start` script is removed - the FastAPI server now serves the build in production.
- **`@types/node` is RETAINED** in `devDependencies` (ISSUE-USR-014) because `vite.config.ts` still imports `path`, uses `__dirname`, and reads `process.env.DISABLE_HMR`. `npm run lint` (`tsc --noEmit`) also needs Node ambient types. Removing `@types/node` would break the lint gate.
- **Dev-server proxy**: in `npm run dev`, requests to `/api/*` are forwarded to `http://localhost:8000` (the FastAPI server). This is configured in `vite.config.ts`'s `server.proxy` block. No `VITE_API_BASE_URL` env var is needed.
- **Vitest globals**: `tsconfig.json` includes `["vitest/globals", "@testing-library/jest-dom"]` so `describe`/`it`/`expect` and jest-dom matchers are typed without imports. `vite.config.ts` sets `test.globals: true` to match. `src/test/setup.ts` loads `@testing-library/jest-dom/vitest` once per test run.
- **Test command** (`npm run test`) is wired to `vitest run`. Batch 7 added test files (`src/api.test.ts`, `src/App.test.tsx`); the command must exit 0 with all tests passing.
- **`npm run lint`** runs `tsc --noEmit` and must pass. `server.ts` was deleted in batch 7; the `tsconfig.json` `exclude` entry for `server.ts` is now a no-op (kept for historical clarity).
- **`npm run build`** runs `vite build` and writes to `dist/`. The output is plain static assets (HTML + JS + CSS); no Node server is bundled.
- **CORS**: the FastAPI server (see `../src/advogado_de_bolso/api.py`) allows `["GET", "POST", "PATCH", "DELETE"]` from configured origins. Frontend code does not need a custom CORS client.
- **TypeScript strict mode** is NOT enabled in `tsconfig.json` (the project ships without `"strict": true`); the `tsc --noEmit` lint gate validates parse + type-resolution only. Batch 7 may add stricter checks if needed.
- **`@vitejs/plugin-react`** is in `dependencies` (not `devDependencies`) to match the pre-batch-6 layout. The duplication is harmless - npm dedupes during install. Batch 7+ may clean this up.

### Adding a new frontend test

1. Create `src/<name>.test.tsx` (or `.test.ts`).
2. Vitest picks it up via the default `include: ["**/*.{test,spec}.?(c|m)[jt]s?(x)"]` pattern.
3. Use the global `describe` / `it` / `expect` from `vitest/globals` - no imports needed.
4. For component tests, import `{render, screen} from '@testing-library/react'` and use `screen.getByRole(...)` etc. The `jest-dom` matchers (`toBeInTheDocument`, `toHaveTextContent`, etc.) are available globally.

### Adding a new API endpoint integration

1. Update `src/api.ts` (batch 7) with a typed wrapper.
2. The Vite dev server will automatically forward `/api/*` calls to `http://localhost:8000` via `server.proxy`. No rebuild needed for dev iteration.
3. In production, the FastAPI server hosts `dist/` and serves the same `/api/*` paths - no proxy needed.

- `npm run build` - `vite build`; must succeed (produces `dist/` consumed by the FastAPI SPA fallback).
- `npm run test` - `vitest run`; batch 7 must report 23 tests passing (`src/api.test.ts` + `src/App.test.tsx`).
- `npm run dev` - Vite dev server on port 5173 with `/api` proxy to port 8000.
- `npm run preview` - serve the production build locally (port 4173 by default).
- `npm run clean` - remove `dist/`.
